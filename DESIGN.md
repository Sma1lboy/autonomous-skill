## Problem Statement

Jackson has ~100+ projects under /Volumes/ssd/i/ and can't actively manage them all. Projects stagnate between work sessions. He wants to cd into any project directory, start Claude Code, run a single command, and have the agent run autonomously — exploring the project, finding issues, fixing them, and iterating on features — continuously within a session budget (default 50 iterations) or until interrupted.

The agent acts as the "project owner" — not a passive executor waiting for instructions, but a proactive contributor that understands the project's goals, the owner's taste, and decides what to work on next.

## What Makes This Cool

The core insight: **skills as constraint > tools as capability**. Instead of giving an agent raw tools and hoping it makes good decisions (the Devin/OpenClaw approach), this agent composes battle-tested gstack workflows (/qa, /review, /investigate, /office-hours) as its decision repertoire. Each workflow encodes human expert judgment. The agent doesn't need to figure out how to do a code review — it invokes /review, which already knows how.

**Terminology**: "the agent" = this autonomous-skill project. "Workflow" or "gstack workflow" = a gstack skill like /qa or /review. "Skill" = a Claude Code skill file (SKILL.md).

The "whoa" moment: you go to sleep, and wake up to a project that's been actively worked on for 8 hours. Not random changes — thoughtful iterations that reflect your priorities and taste.

## Constraints

1. **Platform: Claude Code** — all logic lives as a CC skill (SKILL.md + bash scripts). Not a separate daemon, not a Python wrapper. Chosen because CC's system prompt, permission model, and granularity control are superior to alternatives.
2. **First version: single repo** — runs in the directory it's launched in. Multi-repo coordination is Phase 2.
3. **No auto-merge** — first version creates git branches with proposals, not direct commits to main. Human reviews and merges.
4. **Context window pressure** — CC compresses context over time. The skill must be resilient to this (use files as external memory, not conversation state).
5. **Fork gstack** — gstack skills use AskUserQuestion for human interaction. The autonomous agent needs to intercept these and answer them with its own reasoning. This requires modifying gstack's skill prompts for "autonomous mode."
6. **Iteration cap** — default max 50 iterations per session (configurable). When budget is hit: write summary to log, commit any in-progress branch, exit gracefully. Prevents runaway API costs overnight.
7. **Multiple skill providers** — deferred to Phase 2. Phase 1 hardcodes gstack. The abstraction layer can be added when a second provider actually exists.

## Approaches Considered

### Approach A: Pure Skill — "Single File Flow" (CHOSEN)

A single SKILL.md file + companion bash scripts. All orchestration logic lives in the system prompt. Claude Code executes the loop natively.

```
autonomous-skill/
├── SKILL.md          # Core logic: loop, routing, persona
├── scripts/
│   ├── discover.sh   # Explore project structure, find issues
│   ├── persona.sh    # Build owner persona from git + docs
│   └── loop.sh       # Main loop driver
└── CLAUDE.md
```

**Effort:** S (CC: ~2 hours)
**Risk:** Low
**Pros:** Minimal diff, fastest to ship, stays 100% inside CC ecosystem
**Cons:** No persistent state across sessions, context compression may lose progress, limited task queue sophistication

### Approach B: Skill + Runtime — "Hybrid Architecture"

SKILL.md as entry point, but core logic (task queue, owner persona, skill routing, state persistence) implemented in TypeScript/Bun. SQLite for task queue.

**Effort:** M (CC: ~4 hours)
**Risk:** Medium
**Pros:** State persistence, extensible provider system, proper task queue
**Cons:** Requires build step, more moving parts, overkill for proof of concept

### Approach C: Agent Daemon — "Background Service"

Independent daemon process via Claude Code SDK/API, managed by launchd. True "runs forever" architecture.

**Effort:** L (CC: ~8 hours)
**Risk:** High
**Pros:** True background execution, reliable lifecycle management
**Cons:** High complexity, leaves CC ecosystem, harder to distribute

## Recommended Approach

**Approach A: Pure Skill.** The goal is to prove the core loop works — does the agent make good decisions about what to work on next? Can it effectively compose skills? Does the owner persona produce meaningful prioritization?

If the answer is yes, Approach B becomes the natural evolution (add persistence, proper task queue). If the answer is no, we learn that cheaply without having built infrastructure.

## Core Design: The Autonomous Loop

### 1. Startup Phase

When the skill is invoked:

1. **Read project context**: CLAUDE.md, README.md, package.json, TODOS.md, recent git log
2. **Build owner persona**: Analyze git history (commit style, focus areas, code patterns), project docs, and any existing persona file to understand the owner's priorities and taste
3. **Initial assessment**: Run /health (if available) to get a baseline quality score
4. **Generate task list**: Combine explicit todos (TODOS.md, TODO comments in code, GitHub issues) with discovered issues (/qa findings, /review findings, /investigate findings)
5. **Prioritize**: Use owner persona to rank tasks by likely owner priority

### 2. Main Loop

The loop is driven by `loop.sh` (a bash while-loop), NOT by CC's conversational model. Each iteration spawns a fresh CC invocation with a focused prompt. This avoids context window exhaustion — each iteration gets a clean context loaded from the external state file.

SLUG is derived as: `basename $(git rev-parse --show-toplevel 2>/dev/null || pwd)`

```
# loop.sh drives this — bash while-loop, not CC conversational loop
iteration=0
max_iterations=${MAX_ITERATIONS:-50}  # configurable, default 50

WHILE iteration < max_iterations AND no SIGINT received:
  1. READ state from ~/.gstack/projects/SLUG/autonomous-state.json
     (task list, current progress, owner persona path)
  2. SELECT highest-priority task from task list
  3. ROUTE to appropriate gstack workflow or direct action:
     - Code quality issues → /review, /qa
     - Bugs/errors → /investigate
     - Missing tests → direct action (agent writes tests)
     - Feature design → /office-hours (design only, no implementation)
     - Security → /cso
     - Documentation → /document-release
     NOTE: /ship is EXCLUDED from autonomous mode (violates no-auto-merge)
  4. CREATE branch auto/[task-slug]-[iteration] (timestamp avoids collisions)
  5. SPAWN CC via: claude -p "..." --output-format json > /tmp/auto-result-$iteration.json
     Prompt includes: task description, owner persona, workflow name, target branch.
     CC writes code changes to the checked-out branch. On exit, loop.sh reads
     the JSON output for success/failure status.
     **State handoff contract**: CC writes to git (branch commits) and stdout (JSON result).
     loop.sh reads the JSON result and git status to determine outcome. No shared
     mutable state file during execution — only loop.sh writes to autonomous-state.json.
  6. VERIFY result: loop.sh runs `git diff --stat`, test command (from package.json), build check
  7. IF success (tests pass, build passes):
     - Append to ~/.gstack/projects/SLUG/autonomous-log.jsonl
     - Mark task done in autonomous-state.json
  8. IF failure:
     - git checkout main && git branch -D auto/[task-slug]-[iteration] (clean up broken branch)
     - Log failure in state file (increment strike count)
     - 3 strikes on same task → skip permanently, log reason
     - NOTE: strike counts survive task list refresh (stored by task hash in state file)
  8. REFRESH task list every 5 iterations (re-scan for new issues)
  9. INCREMENT iteration counter

ON EXIT (max iterations, SIGINT, or error):
  - Wait for current CC invocation to finish (do not kill mid-task)
  - IF current branch has uncommitted changes AND tests pass:
      git commit -m "WIP: autonomous partial — [task description]"
    ELSE IF tests fail:
      git stash (preserve work without committing broken state)
  - Write final summary to autonomous-log.jsonl
  - Write exit reason to autonomous-state.json
  - Return to main branch
  - Exit cleanly
```

**Interrupt handling**: loop.sh traps SIGINT. On interrupt: finish current CC invocation (do not kill mid-task), write state, commit any partial branch, exit. A sentinel file (`~/.gstack/projects/SLUG/.stop-autonomous`) also triggers graceful shutdown — useful for remote stop.

**"Direct action" fallback**: When no gstack workflow matches, the agent reads relevant files, writes a patch, and commits to a feature branch using its own judgment. The prompt includes the owner persona for style guidance.

### 3. Owner Persona

**Phase 1 (proof of concept):** Manual `OWNER.md` file in the project root. Simple template:

```markdown
# Owner Persona
## Priorities (what matters most)
## Style (code conventions, commit style)
## Avoid (things NOT to change)
## Current focus (what I'm working on right now)
```

If no `OWNER.md` exists, persona.sh auto-generates a draft from:
- `git log --oneline -50` (recent focus areas, commit style)
- `CLAUDE.md` (explicit preferences)
- `README.md` (project goals)

The auto-generated draft is written to `OWNER.md` for the owner to review and edit. The agent uses whatever is in `OWNER.md` — the owner has final say.

**Phase 2:** Auto-learning that updates OWNER.md based on which branches the owner merges vs. rejects.

Used for:
- **Task prioritization**: "Would the owner care about this?"
- **Decision making**: When gstack workflows ask questions, answer as the owner would
- **Style matching**: Code changes match the owner's coding style

### 4. Skill Routing

The router maps task types to gstack workflows:

| Task Type | Gstack Workflow | Fallback (Direct Action) |
|-----------|----------------|--------------------------|
| Code quality | /review | Agent reads files, writes patch |
| Bugs/errors | /investigate | Agent debugs and patches |
| Test coverage | /qa | Agent writes tests directly |
| Feature design | /office-hours | Agent writes design doc |
| Security | /cso | Agent reviews and patches |
| Documentation | /document-release | Agent writes docs |
| Ship/deploy | **EXCLUDED** | **EXCLUDED** (no-auto-merge) |

**EXCLUDED**: /ship is not available in autonomous mode. The agent can prepare branches for shipping, but the human decides when to merge and deploy.

When no workflow matches, the agent proceeds directly: reads relevant files, reasons about the fix, writes a patch, and commits to an `auto/` branch. The OWNER.md persona guides style and scope decisions.

### 5. Safety Guards

- **No auto-merge**: All changes go to `auto/` feature branches. Human merges.
- **Iteration cap**: Default 50 iterations per session. Configurable via `MAX_ITERATIONS` env var. On cap: graceful shutdown with summary.
- **3-strike rule**: Same task fails 3 times → skip permanently, log reason
- **Scope boundary**: Only touch files within the project directory
- **No destructive operations**: Never rm -rf, never force push, never drop tables
- **Progress logging**: Every action appended to `~/.gstack/projects/SLUG/autonomous-log.jsonl`
- **Interrupt handling**: loop.sh traps SIGINT. Finishes current task, writes state, exits. Sentinel file `.stop-autonomous` also triggers graceful shutdown.
- **No /ship in autonomous mode**: Agent cannot merge, deploy, or push to main

### 6. Gstack Fork Strategy

Fork gstack and modify workflow prompts to support "autonomous mode":

**Prerequisite (Next Steps #0):** Audit gstack's AskUserQuestion usage patterns. If AskUserQuestion is inline in each workflow's prompt, the first step is to refactor it into a centralized hook or decision point. This determines the fork's diff surface.

- Add `AUTONOMOUS_MODE` environment variable that workflows can check
- In autonomous mode, AskUserQuestion calls are replaced with a decision function that reads OWNER.md + context to choose the recommended option
- **Accept divergence**: the fork will diverge from upstream gstack. Periodically cherry-pick upstream improvements, but don't try to maintain full merge compatibility. The autonomous-mode patches are structural changes, not cosmetic ones.

## Open Questions

1. **Notification**: Should the agent notify the owner (email, Slack, webhook) when it completes significant work or hits a blocker? (Nice-to-have for Phase 2)
2. **Multi-branch strategy**: If the agent creates 10 branches overnight, how should they be organized? Naming convention `auto/[task-type]-[description]` is a start, but should it attempt to rebase its own branches?
3. **CC invocation model**: Each loop iteration spawns a fresh CC instance via `claude --print` or similar. Need to validate that this works reliably for 50+ sequential invocations without accumulating zombie processes or rate limit errors.

## Success Criteria

1. Agent can run unattended for 1+ hours without crashing or getting stuck in a loop
2. Produces at least 3 git branches with behavior-changing improvements (bug fixes, new tests, features — not formatting or cosmetic changes)
3. Owner persona accurately reflects project priorities (validated by owner review)
4. Each branch is self-contained and reviewable — no cross-branch dependencies
5. Owner reviews branches and marks at least 2 as "worth merging" or "good direction"
